## Windows Process Tree — Root Cause Analysis

### Failure Classification: F5 (Job Object Assignment Race)

**Root cause**: ProcessManager calls `cmd.spawn()` (process starts running immediately), then later calls `ProcessTreeGuard::attach()` (creates Job Object and assigns root via `AssignProcessToJobObject`). Between these two calls, the fixture root process can spawn child processes that **escape** the Job Object — Windows only auto-assigns children created AFTER the parent is assigned to the Job.

**Consequence**: The grandchild process (spawned by the child, which is spawned by the root) may not be in the Job Object. When the Job is terminated, the grandchild survives. The fallback `taskkill /T` also cannot reach it because the intermediate child has already exited (orphaning the grandchild).

**Evidence**: All failure artifacts show `stage: polling_ready` — the fixture wrote `ready.json` with valid PIDs, but the test's `poll_ready_json` timed out waiting for it. Process contention under rapid 200-iteration loops widens the race window.

**Mitigation applied**: 50ms bounded delay in the fixture before spawning children. This gives ProcessManager ~50x the normal FFI latency window to complete `CreateJobObjectW` + `SetInformationJobObject` + `AssignProcessToJobObject`.

**Complete fix**: ProcessManager should use `CREATE_SUSPENDED` for the root process, assign it to the Job Object while suspended, then call `ResumeThread`. This eliminates the race window entirely. This is a production code change deferred to a future milestone.
