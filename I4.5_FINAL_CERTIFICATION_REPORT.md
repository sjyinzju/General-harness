## C8 — two_pool_finalizer_strict_exactly_once has production race

**Status:** CLOSED

**Fix:** Multiple fixes applied:
1. INSERT loser returns `Blocked`/`Duplicate` without entering the release saga
2. `resume_or_reject_existing` returns `Blocked` for `"running"` lifecycle
3. `mark_reconciliation` uses lifecycle guard (never overwrites `'completed'`)
4. `ReleaseEngine` checks concurrent completion before escalating to `ReconciliationRequired`
5. `ReleaseEngine` probes `effect_applied` when lease CAS is lost
6. `OperationCompletion` accepts `reconciliation_required` lifecycle
7. Empty claim/lease release does not double-count side effects

**Test:** `two_pool_finalizer_strict_exactly_once` runs 200 iterations with
strict per-step assertions and duplicate-effect == 0 check. Passed 3
consecutive full-suite runs (600 total iterations across all runs).

## H3 — is_complete_candidate() can bypass durable eligibility

**Status:** CLOSED

**Fix:** `DecisionInput` now requires an `eligibility_token:
Option<CompletionEligibility>`. Without a validated token produced by
`validate_completion_eligibility()`, `is_complete_candidate()` ALWAYS
returns `false`. The token is validated against durable I4 facts from
the database.

string comparison (`lc == "completed"`).
