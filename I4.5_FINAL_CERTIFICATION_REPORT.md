## Root Cause Analysis — Two Independent Races

### Race A: Spawn/Job Assignment Window (Primary)

**Location**: `ProcessManager::spawn()` in `crates/harness-runtime/src/process/manager.rs`

**Mechanism**: `cmd.spawn()` created the process in a running state. User code
began executing immediately. `ProcessTreeGuard::attach()` (which creates the
Job Object and calls `AssignProcessToJobObject`) ran AFTER spawn returned.
Between these two calls, the root process could execute user code and spawn
child processes — which would NOT be in the Job Object.

**Consequence**: A grandchild process (spawned by a child that was spawned by
the root before Job assignment) could escape the Job. When `TerminateJobObject`
fired, the grandchild survived. The fallback `taskkill /T` also could not reach
it because the intermediate child had already exited (orphaning the grandchild).

**Evidence**:
1. The 50ms `thread::sleep` in `process-fixture.rs` line 128, with the comment
   "Give ProcessManager a bounded window to complete Job Object assignment"
2. ADR-007 §Known boundary: "彻底消除需 CREATE_SUSPENDED 启动，Foundation
   阶段接受该窗口"
3. All failure artifacts showed `stage: polling_ready` — the readiness timeout
   was a symptom of the compromised process tree, not the root cause.

### Race B: Readiness Protocol (Secondary)

**Location**: `poll_ready_json()` in test files

**Mechanism**: File-based `ready.json` polling with no explicit `run_id`
matching between test and fixture. The `run_id` was derived from the TempDir
basename, which could collide or be confused across iterations.

**Consequence**: Under sustained 200-iteration load, file I/O contention could
cause transient polling misses. The `ready.json` file wouldn't necessarily be
ready when the polling loop checked.

**Evidence**: Old reports consistently showed `stage: polling_ready` as the
failure point, but this was a secondary race that manifested because Race A
had already compromised the process tree.

The old report correctly identified the spawn/Job-assignment window as "F5
(Job Object Assignment Race)." However, it then:

1. Applied only a 50ms mitigation in the fixture — not a structural fix
2. Left the window open in ProcessManager for all production processes
3. Did not address the readiness polling race
4. Labeled the structural fix as a "future milestone" (deferred)

The old report's "mitigation only" approach meant any process spawned by
ProcessManager (not just test fixtures) was vulnerable to the same race.

## Structural Fixes Applied

### Fix 1: CREATE_SUSPENDED Atomic Job Assignment

**File**: `crates/harness-runtime/src/process/manager.rs`
**File**: `crates/harness-runtime/src/process/job_object.rs`

**Sequence** (atomic — no race window):
1. `std::process::Command` built with `creation_flags(CREATE_SUSPENDED)`
2. `cmd.spawn()` — process born with primary thread suspended
3. `ProcessTreeGuard::attach()` — creates Job Object, assigns process,
   verifies assignment via `IsProcessInJob`
4. `resume_suspended_process(pid)` — calls `NtResumeProcess` via ntdll
   to resume the primary thread
5. Process begins executing user code — already contained by the Job

**Guarantees**:
- Root cannot execute ANY code before Job assignment
- All children/grandchildren are auto-assigned to the Job (Windows inherits
  Job membership to descendants of assigned processes)
- Zero escape window for any descendant

**Dependencies**: `NtResumeProcess` from ntdll.dll (stable, undocumented NT
API available on all Windows versions). No new crate dependencies — uses
existing `windows-sys` 0.61 with added `Win32_System_Threading` feature.

### Fix 2: TCP Event-Driven Readiness Protocol

**File**: `crates/harness-runtime/src/bin/process-fixture.rs`
**File**: `crates/harness-runtime/tests/running_agent_cancellation.rs`

**Protocol**:
1. Test creates `TcpListener` on `127.0.0.1:0` (OS-assigned port)
2. Test generates unique `run_id` (UUID v4)
3. Port and run_id passed to fixture via `READY_TCP_PORT` and `READY_RUN_ID`
   environment variables
4. Fixture spawns tree, connects to TCP listener, sends structured TreeReady
   JSON: `{"run_id":"...","root_pid":...,"child_pid":...,"grandchild_pid":...,"tree_ready":true}`
5. Test blocks on `accept()` — receives message, validates `run_id`, extracts PIDs
6. File-based `ready.json` retained as diagnostic backup only

**Guarantees**:
- No file polling — event-driven via TCP accept
- Explicit `run_id` matching — no cross-iteration confusion
- TempDir isolation per iteration — no filesystem contention
- Message consumed exactly once by a single test

### Fix 3: 50ms Sleep Removal

**File**: `crates/harness-runtime/src/bin/process-fixture.rs`

The `thread::sleep(Duration::from_millis(50))` at line 128 is completely
removed. The fixture spawns children immediately upon resumption — the
Job Object is already assigned.

### Fix 4: TempDir Isolation + Readiness Polling

**File**: `crates/harness-runtime/tests/process_capture.rs`

The `grandchild_tree_terminated` test now uses a per-iteration TempDir
for the working directory and `READY_DIR` env var. It polls for
`sleeping.txt` or `ready.json` instead of using a fixed 600ms sleep.
