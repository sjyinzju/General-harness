    24 ## 一、第一性原理：核心价值与设计哲学
    25
    26 ### 1.1 这个构想解决的根本问题
    27
    28 当前多 Agent 协作的**核心瓶颈**不是模型能力不足，而是：
    29
    30 | 问题 | 根因 | Harness 的解决方式 |
    31 |------|------|-------------------|
    32 | 上下文过载 → "大海捞针"性能下降（记忆衰退+幻觉） | 单一 Agent 承载所有细节 | **上下文隔离**：总 Agent 只保留 DAG + 状态摘要 |
    33 | 并行效率低 | Agent 串行执行 | **并行调度**：多 Agent 同时在不同 worktree 工作 |
    34 | 质量不可控 | 自我审查不可靠 | **对抗性验证**：独立 verifier 审查每个产出 |
    35 | 长时间偏离目标 | 模型"忘记"初始目标 | **Goal Contract + 状态外置** |
    36
    37 ### 1.2 设计哲学：80% 确定性 + 20% LLM
    38
    39 ```
    40           确定性软件系统 (80%)               LLM 管理能力 (20%)
    41      ┌─────────────────────────┐    ┌──────────────────────────┐
    42      │ • 状态机转换             │    │ • 需求理解与澄清           │
    43      │ • DAG 依赖计算           │    │ • 架构设计与任务拆解        │
    44      │ • 子进程生命周期管理      │    │ • 代码实现                │
    45      │ • Git worktree 隔离      │    │ • 代码审查与语义理解        │
    46      │ • 权限与预算硬限制        │    │ • 调试与根因分析           │
    47      │ • 确定性验收（测试/lint） │    │ • 模糊决策（歧义时）       │
    48      │ • 日志与审计追踪          │    │ • 重规划建议              │
    49      └─────────────────────────┘    └──────────────────────────┘
    50 ```
    51
    52 **核心原则**：总 Agent 不应保存所有上下文、不应自己决定所有状态变化、不应仅根据自然语言判断任务完成。LLM 可以**建议**状态转换，但真正的状态变更由确
       定性规则执行。

    98 ## 三、Q1: 应该采取什么形式？
    99
   100 ### 3.1 结论：独立 Harness（不是斜杠命令或 Skill）
   101
   102 **绝不能**把这个中枢系统写成 Claude Code 里的一个 Slash Command 或 Skill，否则总 Agent 会受限于 Claude 封闭系统的底层逻辑和上下文管理机制。
   103
   104 ### 3.2 推荐架构
   105
   106 ```
   107                     ┌──────────────────────────┐
   108                     │  用户入口（多前端）         │
   109                     │  Claude / Codex / IDE     │
   110                     │  独立 CLI / 桌面 TUI      │
   111                     │  Web UI / MCP Client      │
   112                     └────────────┬─────────────┘
   113                                  │
   114                                  ▼
   115                     ┌──────────────────────────┐
   116                     │   Local Harness Daemon    │
   117                     │   （独立进程，最高权限调度者）│
   118                     └────────────┬─────────────┘
   119                                  │
   120          ┌───────────────────────┼───────────────────────┐
   121          ▼                       ▼                       ▼
   122 ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
   123 │ 目标与计划服务    │   │ 调度器与状态机    │   │ 权限与策略引擎    │
   124 │ - Goal Contract │   │ - DAG 执行      │   │ - 预算控制       │
   125 │ - 需求澄清       │   │ - 检查点         │   │ - 文件访问边界    │
   126 │ - 任务拆解       │   │ - 断点续传       │   │ - 命令白名单     │
   127 │ - 架构设计       │   │ - 并行调度       │   │ - 审批门         │
   128 └────────┬────────┘   └────────┬────────┘   └────────┬────────┘
   129          │                     │                       │
   130          └─────────────────────┼───────────────────────┘
   131                                ▼
   132                     ┌──────────────────────────┐
   133                     │   Agent Adapter Registry  │
   134                     │   （可插拔适配器层）        │
   135                     └────────────┬─────────────┘
   136                                  │
   137          ┌───────────────────────┼───────────────────────────┐
   138          ▼                       ▼                           ▼
   139 ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────────┐
   140 │  Claude Adapter  │   │  Codex Adapter   │   │  ACP / CLI Adapter  │
   141 │  - Agent SDK     │   │  - SDK           │   │  - stdio JSON-RPC   │
   142 │  - stream-json   │   │  - app-server    │   │  - JSONL            │
   143 │  - claude -p     │   │  - exec --json   │   │  - PTY (last resort)│
   144 └────────┬────────┘   └────────┬────────┘   └──────────┬──────────┘
   145          │                     │                        │
   146          ▼                     ▼                        ▼
   147 ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────────┐
   148 │  Claude Code     │   │  Codex CLI       │   │  Gemini / Aider     │
   149 │  + 配置的 Model  │   │  + 配置的 Model  │   │  / 自定义 CLI        │
   150 └─────────────────┘   └─────────────────┘   └─────────────────────┘
   151                                  │
   152                                  ▼
   153                     ┌──────────────────────────┐
   154                     │  Git Worktree / Container │
   155                     │  确定性测试 / Lint / Build │
   156                     └──────────────────────────┘
   157 ```
   158
   159 ### 3.3 三种形式的具体定位
   160
   161 **独立 Harness Daemon（主体）**
   162
   163 拥有：项目状态、子进程生命周期、任务队列、Git 工作区、权限与预算、日志与恢复、Agent 路由、最终验收。
   164
   165 **斜杠命令（入口，不是主体）**
   166
   167 ```
   168 /harness build 一个支持用户登录和支付的 SaaS 后台
   169 /harness status
   170 /harness approve plan
   171 /harness cancel
   172 ```
   173
   174 斜杠命令内部只调用本地 Harness API，不能承担真正的长时间编排。它只是一个薄薄的转发层。
   175
   176 **Skill（使用说明和接入层）**
   177
   178 告诉 Claude Code 或 Codex 如何提交任务、查询状态、读取任务单、提交结构化结果。Skill 本身不适合持有可靠队列、进程租约和全局状态。
   179
   180 **Harness 作为 MCP Server（可选）**
   181
   182 暴露工具：
   183 - `harness.create_project`
   184 - `harness.get_status`
   185 - `harness.list_tasks`
   186 - `harness.approve`
   187 - `harness.cancel`
   188 - `harness.submit_result`
   189
   190 这样 Claude、Codex 或其他支持 MCP 的前端都可以作为入口。MCP 只是控制接口，真正状态仍在 Harness 进程和数据库中。

   194 ## 四、Q2: 如何获取本地 Agent 信息并实现 Headless 交互？
   195
   196 ### 4.1 Agent 发现：Runtime Profile，不只是 API Key 扫描
   197
   198 **不应假设**"Claude Code + 任意 LLM"或"Codex + 任意 Provider"。
   199
   200 **应表示为**："Claude Code + 当前配置的 Provider + 当前 Model + 当前认证状态 + 当前权限 + 实际能力探测结果"。
   201
   202 ### 4.2 Runtime Profile 数据结构
   203
   204 ```json
   205 {
   206   "id": "codex-gpt-main",
   207   "agent": "codex",
   208   "agentVersion": "x.y.z",
   209   "adapter": "codex-app-server",
   210   "provider": "openai",
   211   "model": "gpt-4",
   212   "authState": "verified",
   213   "workspaceModes": ["read", "write", "shell"],
   214   "structuredOutput": true,
   215   "streaming": true,
   216   "resumeSession": true,
   217   "sandbox": "workspace-write",
   218   "maxConcurrency": 2,
   219   "status": "AVAILABLE"
   220 }
   221 ```
   222
   223 ### 4.3 发现流程
   224
   225 ```
   226 Step 1: 扫描可执行程序
   227 ├─ where/which claude, codex, gemini, aider
   228 ├─ PATH 环境变量
   229 ├─ npm 全局包
   230 ├─ pip/uv 工具目录
   231 ├─ Cargo bin
   232 ├─ Homebrew / Scoop / Chocolatey
   233 ├─ 常见安装目录
   234 └─ 用户显式注册的路径
   235
   236 Step 2: 每个 Adapter 执行安全能力探测
   237 ├─ detect()           → 是否可执行
   238 ├─ getVersion()       → 版本号
   239 ├─ inspectConfiguration() → 读取配置引用（不读密钥值）
   240 │    ├─ provider 名称
   241 │    ├─ base URL
   242 │    ├─ model ID
   243 │    ├─ 认证方式（login / env_var / keychain）
   244 │    ├─ sandbox 和审批配置
   245 │    └─ 配置文件位置
   246 ├─ checkAuthentication() → Agent 自身的认证状态接口
   247 ├─ listAvailableProfiles() → 可用运行配置列表
   248 └─ probe()            → 微型任务验证
   249      ├─ 只读仓库结构总结
   250      ├─ 创建文件（在临时仓库）
   251      ├─ 执行测试
   252      ├─ 输出指定 JSON Schema
   253      ├─ 中断并恢复
   254      └─ 达到预算后停止
   255
   256 Step 3: 状态分级
   257 ├─ DETECTED      找到了可执行程序
   258 ├─ CONFIGURED    找到了有效配置
   259 ├─ AUTHENTICATED Agent 确认认证有效
   260 ├─ PROBED        完成最小能力测试
   261 ├─ AVAILABLE     可进入调度池
   262 └─ DEGRADED      部分能力不可用
   263
   264 Step 4: 能力缓存与刷新
   265 ├─ 结果写入 agents_registry.json
   266 ├─ 定时重新探测（可配置间隔）
   267 └─ 调度失败时触发即时重新探测
   268 ```
   269
   270 ### 4.4 安全核心原则
   271
   272 - 探测命令必须设置超时
   273 - 不修改用户文件（在临时目录探测）
   274 - 不发送敏感配置到网络
   275 - 不在日志中记录密钥
   276 - 只读取配置的"引用"（provider 名称、model ID），**不读取密钥值**
   277 - API Key 永远不进入 Harness 数据库
   278
   279 ### 4.5 无头交互（Headless Execution）
   280
   281 **绝不能**通过模拟键盘输入或打开新终端窗口来交互——极易崩溃且无法准确捕获报错。
   282
   283 四种交互方式，按优先级排列：
   284
   285 ```
   286 优先级 1: 官方 SDK 或 Agent Server（最稳定）
   287 ─────────────────────────────────────────────
   288 Claude Code:
   289   - Claude Agent SDK（与 CLI 相同能力的 Agent Loop）
   290   - claude -p --input-format stream-json --output-format stream-json
   291   - 支持 JSON Schema、最大轮数、最大预算
   292
   293 Codex:
   294   - Codex SDK
   295   - Codex app-server（stdio JSONL / WebSocket / Unix socket）
   296   - codex exec --json（JSONL 输出）
   297   - 支持双向 JSON-RPC（会话、审批、流式事件）
   298
   299 优先级 2: ACP（Agent Communication Protocol）
   300 ─────────────────────────────────────────────
   301   - 支持的 Agent 作为后台子进程启动
   302   - 通过 stdio JSON-RPC 与 Harness 交互
   303   - 不需要终端窗口
   304   - 目标：将编码 Agent 与客户端解耦
   305
   306 优先级 3: Headless CLI
   307 ─────────────────────────────────────────────
   308   - codex exec --json "任务"
   309   - claude -p --output-format stream-json "任务"
   310   - gemini -p "任务" --output-format json
   311   - Harness 使用子进程 stdin/stdout/stderr/exit code/signal
   312   - Windows 中隐藏窗口，用 Job Object 管理子进程树
   313
   314 优先级 4: PTY 模拟终端（最后手段）
   315 ─────────────────────────────────────────────
   316   - 仅当 Agent 没有 SDK、协议或无交互模式时
   317   - 自动输入命令、解析屏幕文本、模拟按键、检测提示
   318   - 极其脆弱，CLI 界面一更新就可能失效
   319   - 只能作为兼容层，不能作为核心架构
   320 ```
   321
   322 ### 4.6 多 Agent 并行操作同一项目的隔离方案
   323
   324 **绝不能让多个写入型 Agent 同时在同一个工作目录中自由修改文件。** 这是整个系统中比提示词设计更重要的问题。
   325
   326 ```
   327 project/
   328 ├── main worktree（主分支，只读参考）
   329 └── worktrees/
   330     ├── task-auth-codex/     ← Codex 独占写权限
   331     ├── task-ui-claude/      ← Claude 独占写权限
   332     └── task-tests-gemini/   ← Gemini 独占写权限
   333 ```
   334
   335 **每个任务拥有：**
   336 - 独立 Git worktree
   337 - 独立分支
   338 - 独立进程
   339 - 独立依赖缓存或容器
   340 - 明确允许修改的文件范围（scope.allowedPaths）
   341 - 一个 workspace lease（租约，防止多 Agent 冲突）
   342
   343 **完成后由 Integrator：**
   344 - 检查提交
   345 - 运行测试
   346 - 合并或 cherry-pick
   347 - 解决冲突
   348 - 在集成分支再次验证
   349
   350 **规则**：只读分析 Agent 可以共享仓库快照；写入 Agent 必须隔离。

   354 ## 五、Q3: 状态机设计 —— 如何防止偏离轨道和幻觉？
   355
   356 ### 5.1 长流程任务的最大敌人："熵增"
   357
   358 上下文混乱 → 模型忘记初始目标 → 幻觉累积。解决方案是**状态外置 + 严格的阶段校验 + 动态上下文管理**。
   359
   360 ### 5.2 双层状态机
   361
   362 #### 第一层：项目级状态机
   363
   364 ```
   365                     ┌─────────┐
   366                     │ CREATED │
   367                     └────┬────┘
   368                          ↓
   369                     ┌───────────┐
   370                     │ CLARIFYING │ ← 需求澄清、与用户对齐
   371                     └────┬──────┘
   372                          ↓
   373                     ┌─────────────┐
   374                     │ GOAL_LOCKED  │ ← 生成 Goal Contract，此后不可随意修改
   375                     └────┬────────┘
   376                          ↓
   377                     ┌───────────────┐
   378                     │ ENV_DISCOVERY  │ ← 扫描 Agent 能力、仓库结构
   379                     └────┬──────────┘
   380                          ↓
   381                     ┌──────────┐
   382                     │ PLANNING │ ← 架构设计 + 任务 DAG
   383                     └────┬─────┘
   384                          ↓
   385                     ┌──────────────┐
   386                     │ PLAN_REVIEW  │ ← 内部审查 DAG 合理性
   387                     └────┬─────────┘
   388                          ↓
   389               ┌──────────┴──────────┐
   390               │ 有歧义/高风险？       │
   391               ▼                     ▼
   392      ┌─────────────────┐   ┌──────────────────┐
   393      │NEEDS_USER_APPROVAL│  │   SCHEDULING     │
   394      │（等待用户确认 plan）│  │ （开始执行调度）  │
   395      └────────┬────────┘   └────────┬─────────┘
   396               │                     │
   397               └──────────┬──────────┘
   398                          ↓
   399                     ┌───────────┐
   400                     │ EXECUTING │ ← 并行分派任务给子 Agent
   401                     └────┬──────┘
   402                          ↓
   403                     ┌─────────────┐
   404                     │ INTEGRATING │ ← 合并 worktree，解决冲突
   405                     └────┬────────┘
   406                          ↓
   407                     ┌───────────┐
   408                     │ VERIFYING │ ← 端到端验收
   409                     └────┬──────┘
   410                          │
   411          ┌───────────────┼───────────────┐
   412          ▼               ▼               ▼
   413     ┌─────────┐   ┌──────────┐   ┌──────────┐
   414     │  pass   │   │ fixable  │   │ blocked  │
   415     └────┬────┘   └────┬─────┘   └────┬─────┘
   416          ▼             ▼              ▼
   417     ┌──────────┐ ┌──────────┐  ┌──────────┐
   418     │DELIVERING│ │REPAIRING │  │ BLOCKED  │
   419     └────┬─────┘ └──┬───┬───┘  └──────────┘
   420          ▼          │   │
   421     ┌────────┐      │   └──────────┐
   422     │  DONE  │      ▼              ▼
   423     └────────┘ ┌──────────┐  ┌──────────────┐
   424                │EXECUTING │  │PLAN_REVISION │
   425                │（重新执行）│  │（重新规划）   │
   426                └──────────┘  └──────────────┘
   427 ```
   428
   429 #### 第二层：任务级状态
   430
   431 ```
   432 PENDING ──→ READY ──→ LEASED ──→ RUNNING ──→ SUBMITTED
   433                                                 │
   434                                                 ▼
   435                                 ┌─────── REVIEWING ───────┐
   436                                 │                          │
   437                                 ▼                          ▼
   438                            VERIFIED                  FAILED_RETRYABLE
   439                                 │                          │
   440                                 ▼                          ▼
   441                               MERGED              FAILED_TERMINAL
   442                                                       │
   443                                                       ▼
   444                               ┌─────────────→ BLOCKED（需人工介入）
   445                               │
   446                               ▼
   447                           SUPERSEDED（被更高优先级任务取代）
   448                               │
   449                               ▼
   450                           CANCELLED
   451 ```
   452
   453 ### 5.3 状态转换必须由代码决定
   454
   455 **错误方式**：总 Agent "觉得差不多完成了"，所以进入下一阶段。
   456
   457 **正确方式**：
   458
   459 ```
   460 任务进入 VERIFIED 状态，必须同时满足 ALL 条件：
   461
   462 ✅ Agent 提交了符合 Schema 的结果
   463 ✅ 有可定位的 Git commit 或 artifact hash
   464 ✅ 必需命令退出码为 0（测试、lint、build）
   465 ✅ 验收检查全部通过
   466 ✅ 没有未解决的 blocker
   467 ✅ 独立 verifier 给出通过结果
   468
   469 LLM 可以建议：
   470   "建议进入 VERIFYING"
   471   "建议重新规划"
   472   "建议把任务交给另一 Agent"
   473
   474 但真正的状态转换由确定性规则执行。
   475 ```
   476
   477 ### 5.4 Goal Contract（目标契约）
   478
   479 项目开始后生成的不可修改的目标锚定文件：
   480
   481 ```json
   482 {
   483   "projectId": "proj-2026-0714-001",
   484   "objective": "构建一个本地多 Agent 编排系统",
   485   "deliverables": [
   486     "可运行 CLI",
   487     "Claude 和 Codex Adapter",
   488     "状态持久化",
   489     "并行 worktree 支持"
   490   ],
   491   "acceptance": [
   492     "进程重启后可以恢复项目",
   493     "两个 Agent 可并行完成独立任务",
   494     "最终集成测试通过",
   495     "用户可查看任务和成本"
   496   ],
   497   "constraints": [
   498     "不读取或存储 API Key 明文",
   499     "不允许 Agent 直接修改主分支",
   500     "高风险命令必须审批",
   501     "单任务最大预算 $4"
   502   ],
   503   "non_goals": [
   504     "第一版不支持所有 Agent"
   505   ],
   506   "createdAt": "2026-07-14T10:00:00Z",
   507   "version": 1
   508 }
   509 ```
   510
   511 **关键规则**：任何重新规划都必须说明它改变了哪些目标、约束或交付物，并形成版本化的 Plan Revision。Goal Contract 不可被 LLM 单方面修改。
   512
   513 ### 5.5 四层上下文架构（反幻觉核心）
   514
   515 总 Agent **不保存完整对话上下文**：
   516
   517 ```
   518 ┌─────────────────────────────────────────────┐
   519 │ L0: 不可变 Goal Contract                    │
   520 │     永远注入总 Agent 系统提示                │
   521 │     大小: ~500 tokens                       │
   522 ├─────────────────────────────────────────────┤
   523 │ L1: 当前项目状态快照                         │
   524 │     - 当前阶段                               │
   525 │     - 未完成任务列表（仅标题+状态）            │
   526 │     - 最近状态变化（最近 5 条）               │
   527 │     - 当前风险列表                           │
   528 │     大小: ~1000 tokens                       │
   529 ├─────────────────────────────────────────────┤
   530 │ L2: 当前焦点任务的 Task Dossier              │
   531 │     - 任务目标                               │
   532 │     - 输入 artifact 引用                     │
   533 │     - 验收条件                               │
   534 │     - 上一步结论的摘要（不是完整日志）          │
   535 │     大小: ~2000 tokens                       │
   536 ├─────────────────────────────────────────────┤
   537 │ L3: 完整证据（外部存储，不进入上下文）         │
   538 │     - Agent 完整 transcript                  │
   539 │     - 测试输出                               │
   540 │     - Git diff                               │
   541 │     - 文件内容                               │
   542 │     按需通过工具查询                          │
   543 └─────────────────────────────────────────────┘
   544 ```
   545
   546 **每次总 Agent turn 只看到 L0+L1+L2（~3500 tokens）+ 当前决策请求**，而不是把所有 Agent 的几万行对话塞进上下文。
   547
   548 ### 5.6 结构化任务信封
   549
   550 每个 Worker 收到的不是自由文本，而是结构化信封：
   551
   552 ```json
   553 {
   554   "projectGoal": "构建一个本地多 Agent 编排系统",
   555   "taskId": "TASK-014",
   556   "taskGoal": "实现 OAuth 回调处理",
   557   "scope": {
   558     "allowedPaths": ["src/auth/**", "tests/auth/**"],
   559     "forbiddenPaths": ["infra/**"]
   560   },
   561   "dependencies": ["TASK-009"],
   562   "inputs": [
   563     { "type": "artifact", "path": "design/auth-design.md" },
   564     { "type": "artifact", "path": "src/auth/models.ts" }
   565   ],
   566   "acceptanceChecks": [
   567     "npm test -- auth",
   568     "npm run lint"
   569   ],
   570   "allowedTools": ["read", "write", "edit", "bash"],
   571   "outputSchema": "TaskResultV1",
   572   "budget": {
   573     "maxTurns": 20,
   574     "maxCost": 4
   575   }
   576 }
   577 ```
   578
   579 返回也必须结构化：
   580
   581 ```json
   582 {
   583   "status": "completed",
   584   "summary": "实现 OAuth 回调及错误处理",
   585   "gitCommit": "abc123def456",
   586   "changedFiles": [
   587     "src/auth/callback.ts",
   588     "tests/auth/callback.test.ts"
   589   ],
   590   "checks": [
   591     { "command": "npm test -- auth", "exitCode": 0, "output": ".../task-014-test.log" }
   592   ],
   593   "blockers": [],
   594   "risks": [],
   595   "proposedFollowups": ["添加 rate limiting 到回调端点"]
   596 }
   597 ```
   598
   599 ### 5.7 携带证据的完成声明
   600
   601 Agent 口头说"已经完成"不具有任何状态意义。
   602
   603 完成必须携带：
   604 - Git commit hash
   605 - 文件 diff
   606 - 文件 hash
   607 - 执行的测试命令
   608 - exit code
   609 - 测试输出位置
   610 - 生成物路径
   611 - 逐条对应验收条件的结论
   612
   613 ### 5.8 独立验证
   614
   615 实现者和验证者**不是同一个运行配置**：
   616
   617 - Claude 规划 → Codex 实现 → 另一个 Codex/Claude 实例审查 → 确定性测试系统验收
   618
   619 对于主观任务，使用有明确 Rubric 的 LLM Judge；对于代码、构建、文件、API 和测试结果，**必须优先使用确定性检查**。
   620
   621 ### 5.9 限制修复循环
   622
   623 Repair Loop 硬限制：
   624
   625 ```
   626 maxAttempts: 3           ← 最大尝试次数
   627 maxCost: $2              ← 最大修复成本
   628 maxTime: 300s            ← 最大修复时间
   629 sameErrorFingerprint: 2  ← 相同错误指纹出现 2 次 → 中止
   630 noFileChangeDetection: true  ← 无文件变化 → 中止
   631 testNoImprovement: true      ← 测试结果无改善 → 中止
   632 autoSwapAgent: true          ← 自动更换 Agent 尝试
   633 ```
   634
   635 超限后：重新规划或请求用户介入。

   639 ## 六、Q4: Tools、Hooks 与其他构成要素
   640
   641 ### 6.1 核心 Tools（最小原子集）
   642
   643 不需要设计花哨的工具——这是给总 Agent 用的，不是给子 Agent 的：
   644
   645 | 工具 | 用途 | 说明 |
   646 |------|------|------|
   647 | **AgentDiscovery** | 扫描并列出本地可用 Agent 及其能力 | 启动时运行，结果缓存 |
   648 | **AgentDispatch** | 分派任务给指定 Agent 后端 | 核心调度原语 |
   649 | **StateManager** | 读写 Harness 持久化状态 | DAG、检查点、决策日志 |
   650 | **VerificationOrchestrator** | 编排验证流程 | 收集产出 → Reviewer → Debugger 循环 |
   651 | **GoalContractManager** | 创建和版本化 Goal Contract | 任何修改必须经过此工具 |
   652 | **GitWorktreeManager** | 创建/合并/清理 worktree | 任务隔离的基础设施 |
   653 | **CheckpointManager** | 保存和恢复检查点 | 支持断点续传 |
   654 | **BudgetTracker** | 跟踪和强制执行预算限制 | 硬限制，不可绕过 |
   655 | **UserApproval** | 关键决策前请求人类确认 | 不确定就先问 |
   656
   657 ### 6.2 子 Agent 的 Tools（根据任务类型动态授予）
   658
   659 子 Agent 获取最小必要工具集：
   660
   661 ```yaml
   662 read_only_agent: [read, glob, grep, web_search, lsp]
   663 executor_agent: [read, write, edit, bash, notebook_edit]
   664 reviewer_agent: [read, glob, grep, bash, lsp]
   665 debugger_agent: [read, write, edit, bash]
   666 planner_agent: [read, glob, grep, web_search, web_fetch]
   667 ```
   668
   669 **不授予子 Agent 的工具**：AgentDispatch（防止子 Agent 再去调度孙 Agent，导致失控）、StateManager（子 Agent 不应该修改全局状态）。
   670
   671 ### 6.3 Harness 统一 Hooks 系统
   672
   673 不依赖 Claude 或 Codex 自己的 Hooks。Harness 内部定义跨 Agent Hook：
   674
   675 ```yaml
   676 hooks:
   677   # 项目生命周期
   678   on_project_created:
   679     - init_git_repo
   680     - create_harness_dir_structure
   681
   682   # 任务调度
   683   before_dispatch:
   684     - verify_workspace_lease    # 确保没有 worktree 冲突
   685     - snapshot_dependency_state # 记录依赖状态快照
   686     - auto_commit_if_dirty      # 防止子 Agent 改崩代码无法回滚
   687
   688   after_agent_started:
   689     - start_cost_tracker
   690     - start_timeout_timer
   691
   692   # 工具调用（安全拦截层）
   693   before_tool_call:
   694     - check_file_in_scope       # 验证文件在允许范围内
   695     - check_command_whitelist   # 危险命令拦截
   696     - check_budget_remaining
   697
   698   after_tool_call:
   699     - log_tool_usage            # 审计日志
   700     - incremental_cost_update
   701
   702   # 文件变更
   703   after_file_change:
   704     - auto_format               # 自动格式化
   705     - auto_lint                 # 自动 lint 检查
   706
   707   before_commit:
   708     - run_unit_tests
   709     - check_for_secrets         # 密钥扫描
   710
   711   # 任务完成
   712   after_task_submitted:
   713     - validate_output_schema    # 验证输出格式
   714     - compute_artifact_hash
   715     - update_task_status
   716
   717   # 验证
   718   before_verify:
   719     - collect_artifacts
   720     - prepare_test_environment
   721
   722   after_verify:
   723     - update_verification_status
   724     - log_verification_evidence
   725
   726   # 合并
   727   before_merge:
   728     - check_conflicts
   729     - run_integration_tests
   730
   731   after_merge:
   732     - cleanup_worktree
   733     - release_workspace_lease
   734
   735   # 异常处理
   736   on_retry:
   737     - increment_retry_counter
   738     - check_retry_limits         # 超限自动中止
   739
   740   on_stall:                      # Agent 长时间无输出
   741     - send_health_check
   742     - escalate_if_no_response
   743
   744   on_budget_exceeded:
   745     - force_terminate_agent
   746     - notify_orchestrator
   747
   748   on_approval_required:
   749     - pause_dag_execution
   750     - notify_user
   751
   752   on_checkpoint:
   753     - save_full_state_snapshot
   754     - verify_checkpoint_integrity
   755
   756   on_cancel:
   757     - terminate_all_subprocesses
   758     - cleanup_all_worktrees
   759     - save_final_state
   760
   761   # 周期性自检
   762   periodic_health_check:
   763     interval: 5m
   764     - check_all_agents_alive
   765     - check_no_orphan_processes
   766     - check_disk_space

   790 ## 七、Agent 分工与路由策略
   791
   792 ### 7.1 任务类型（不是固定角色）
   793
   794 不要把角色限制为 plan / execute / debug：
   795
   796 | 任务类型 | 职责 | 典型工具 | 输出 |
   797 |---------|------|---------|------|
   798 | **Clarifier** | 把自然语言需求转为 Goal Contract | read, ask_user | Goal Contract JSON |
   799 | **Explorer** | 扫描仓库结构、依赖、已有架构 | read, glob, grep, lsp | 仓库分析报告 |
   800 | **Architect** | 架构设计、任务 DAG、ADR | read, glob, grep, web_search | plan.json + ADR |
   801 | **Implementer** | 完成有边界的代码任务 | read, write, edit, bash | commit + 文件变更 |
   802 | **Tester** | 编写或执行测试 | read, write, bash | 测试报告 |
   803 | **Reviewer** | 代码审查、需求对齐检查 | read, glob, grep | 审查报告 |
   804 | **Security Reviewer** | 权限、依赖、密钥、危险命令 | read, glob, grep, bash | 安全报告 |
   805 | **Debugger** | 根据失败证据定位根因 | read, edit, bash | 修复 commit |
   806 | **Integrator** | 合并 worktree，解决冲突 | bash(git) | 合并后的分支 |
   807 | **Deliverer** | 生成构建物、说明、交付清单 | bash, read | 交付包 |
   808
   809 这些是**任务类型**，不一定永久绑定某个 Agent。同一个 Codex 实例可以执行 Implementer，也可以执行 Reviewer，但**关键任务必须避免自我审查**。
   810
   811 ### 7.2 两阶段 Agent 路由
   812
   813 第一版内置初始经验（Claude 适合架构、Codex 适合执行等），但长期**不能依赖静态偏见**——这些判断会随着版本更新迅速失效。
   814
   815 #### 第一阶段：硬过滤
   816
   817 ```
   818 Agent 是否 AVAILABLE？
   819    ├─ 是否有写权限？（对于写入任务）
   820    ├─ 是否支持目标语言/框架？
   821    ├─ 是否支持结构化输出？
   822    ├─ 是否能运行 shell？
   823    ├─ 是否支持会话恢复？
   824    ├─ 是否满足成本限制？
   825    ├─ 是否能在当前平台运行？
   826    └─ 是否拥有需要的 MCP 工具？
   827 ```
   828
   829 #### 第二阶段：评分
   830
   831 ```
   832 score =
   833     task_affinity           # 任务类型亲和度（初始经验值）
   834   + historical_success_rate # 历史成功率（按任务类型）
   835   + verification_pass_rate  # 验证首次通过率
   836   + reliability             # 可靠性分数（完成率/中断率）
   837   + context_fit             # 上下文匹配度（语言/框架/代码库熟悉度）
   838   - estimated_cost          # 预估成本
   839   - estimated_latency       # 预估延迟
   840   - recent_failure_penalty  # 近期失败惩罚（指数衰减）
   841 ```
   842
   843 ### 7.3 持续学习
   844
   845 Harness 应记录每个 Agent Profile 的：
   846
   847 ```
   848 ├─ 不同任务类型的成功率
   849 ├─ 首次测试通过率
   850 ├─ 平均修复轮数
   851 ├─ 引入回归的次数
   852 ├─ 平均成本
   853 ├─ 平均运行时长
   854 ├─ 用户撤销率
   855 └─ 审查发现的问题数量
   856 ```
   857
   858 因此系统会逐渐学习：
   859
   860 > 在这个用户的这个代码库里，
   861 > Claude-profile-A 更适合规划，
   862 > Codex-profile-B 更适合 TypeScript 实现，
   863 > Codex-profile-C 更适合测试修复。
   864
   865 而不是形成全局、静态的品牌偏见。

   869 ## 八、技术选型：LangGraph、Temporal 还是自研？
   870
   871 ### 8.1 分层建议
   872
   873 ```
   874 ┌─────────────────────────────────────────┐
   875 │ Temporal（外层持久工作流）                │
   876 │ - 数小时/天级长时间运行                   │
   877 │ - 进程崩溃后可靠恢复                      │
   878 │ - 重试、心跳、计时器                      │
   879 │ - 第二/三阶段引入                         │
   880 ├─────────────────────────────────────────┤
   881 │ LangGraph（内层 Agentic 工作流）          │
   882 │ - Planner 和 Reviewer 子图               │
   883 │ - interrupt 用户审批                      │
   884 │ - checkpoint 状态保存                    │
   885 │ - repair 循环                            │
   886 │ - 并行 subgraph                          │
   887 │ - 第一/二阶段引入                         │
   888 ├─────────────────────────────────────────┤
   889 │ 自研 State Machine（核心确定性逻辑）       │
   890 │ - DAG 依赖解析                           │
   891 │ - 状态转换硬规则                          │
   892 │ - 权限与预算强制执行                      │
   893 │ - 第一阶段实现                            │
   894 ├─────────────────────────────────────────┤
   895 │ Agent SDK / CLI（实际执行单元）            │
   896 │ - Claude Agent SDK                       │
   897 │ - Codex SDK / app-server                 │
   898 │ - stream-json / JSON-RPC / ACP           │
   899 └─────────────────────────────────────────┘
   900 ```
   901
   902 ### 8.2 MVP 阶段建议：自研 + SQLite
   903
   904 第一阶段**不引入 LangGraph 或 Temporal**：
   905
   906 - TypeScript（与 Claude Code 和 Codex SDK 同生态）
   907 - SQLite（状态持久化）
   908 - 明确的状态机（硬编码，确保确定性）
   909 - 基于文件系统的本地队列
   910 - Git worktree
   911 - Claude Agent SDK + Codex SDK
   912 - JSON Schema / Zod（结构化输出验证）
   913 - 简单 CLI/TUI
   914
   915 ### 8.3 第二阶段引入 LangGraph
   916
   917 当 Agentic 工作流变复杂时：
   918
   919 - State、Node、Edge 模型与 Harness 的需求天然匹配
   920 - Checkpoint 机制用于保存/恢复状态
   921 - interrupt 用于用户审批门
   922 - 子图用于并行执行
   923
   924 ### 8.4 第三阶段考虑 Temporal
   925
   926 真正需要生产级持久工作流时：
   927 - 进程崩溃恢复
   928 - 分布式执行（多机器）
   929 - 复杂的重试和超时策略

  1041 ## 十、核心设计原则总结
  1042
  1043 ### 10.1 这个 Harness 应当做什么
  1044
  1045 ```
  1046 ✅ 锁定目标，维护 Goal Contract
  1047 ✅ 维护任务 DAG 和依赖关系
  1048 ✅ 选择执行资源（Agent + Model 路由）
  1049 ✅ 控制权限和预算（硬限制）
  1050 ✅ 收集可验证的证据（commit、hash、exit code、test output）
  1051 ✅ 根据确定性状态决定下一步
  1052 ✅ 只在歧义、重规划和风险决策时调用管理 LLM
  1053 ✅ 支持断点续传和进程崩溃恢复
  1054 ✅ 记录完整的审计追踪
  1055 ```
  1056
  1057 ### 10.2 这个 Harness 不应当做什么
  1058
  1059 ```
  1060 ❌ 自己保存所有上下文
  1061 ❌ 自己决定所有状态变化
  1062 ❌ 仅根据自然语言判断任务完成
  1063 ❌ 让 Worker 直接互相自由聊天
  1064 ❌ 让多个 Agent 共享同一个可写工作目录
  1065 ❌ 把所有密钥集中读取
  1066 ❌ 无限执行和无限 Debug
  1067 ❌ 绕过用户的审批门
  1068 ```
  1069
  1070 ### 10.3 一句话总结
  1071
  1072 > 这不是"再造一个更强的总 Agent"，而是**确定性的本地编排运行时 + 少量负责规划与异常决策的管理 LLM + 多个可插拔 Agent Adapter**。它将"多个 Agent 互相
       对话的 Demo"变成真正可以长期运行、恢复、审计和交付成果的**本地 Agent 操作系统**。
